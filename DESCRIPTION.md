Нужно спроектировать и поэтапно реализовать локальное offline-first приложение на Python для хранения и извлечения знаний по reverse engineering артефактам.

Основная цель:
Сделать локальную knowledge base для RE, которую сможет использовать человек и внешний агент через MCP/HTTP API. Система должна позволять сохранять, искать, обновлять и просматривать знания о функциях, структурах и гипотезах по нескольким бинарям и нескольким проектам/workspace. Всё должно работать локально без интернета.

Ключевые ограничения:
- Язык: Python
- Хранилище: SQLite
- Минимум сторонних библиотек
- Все зависимости должны быть такими, чтобы их можно было заранее скачать через pip download и установить offline
- ОС: Windows
- Работа полностью локальная
- Локальный HTTP/MCP допустим
- Простота развёртывания важнее всего
- Сначала backend и MCP, потом GUI
- GUI нужен web-based и пригодный для ежедневной работы
- Никакой обязательной зависимости от облачных сервисов
- Псевдокод целиком хранить не нужно, но нужно иметь возможность сохранять его фрагменты как evidence/attachments

Что должно быть в системе

1. Сущности знания
Минимальные сущности:
- function
- structure
- global hypothesis

Также нужно предусмотреть связанные сущности/артефакты:
- evidence
- tags
- edges/relations между сущностями
- version history для GUI
- audit trail

2. Поддержка нескольких проектов
- Система должна поддерживать несколько проектов/workspaces
- Данные разных проектов не должны смешиваться
- В одном проекте может быть несколько бинарей
- Записи должны быть привязаны к project_id и binary_id
- Для подключения агента нужен отдельный MCP port для каждого проекта
- Архитектура должна быть такой, чтобы позже можно было поддержать и единый сервер с маршрутизацией по project_id

3. Карточка функции
Для функции должны быть обязательные и хорошо структурированные поля:
- function_id
- binary_id
- address
- raw_name / current_name
- summary
- important_variables
- behavior_description
- related_entities
- observed_facts
- hypotheses
- callers
- callees
- used_apis
- strings
- constants
- confidence
- evidence_refs
- tags
- created_at
- updated_at
- created_by
- updated_by

Observed facts и hypotheses должны храниться раздельно.
Для hypotheses нужен статус:
- new
- probable
- confirmed
- rejected

4. Evidence
Нужно хранить evidence:
- привязка к сущности
- тип evidence
- адреса / диапазоны адресов / xrefs / блоки
- текстовое описание
- опционально фрагменты псевдокода или других данных
- вложения как отдельные файлы допустимы
- ссылки на файлы должны храниться в SQLite

5. История и аудит
- Нужна история версий для просмотра в GUI
- История нужна человеку, но не должна засорять ответы агенту
- Нужен audit trail: кто изменил запись — пользователь или агент
- У каждой записи должен быть source/origin
- Не нужно реализовывать сложный merge нескольких версий одной и той же функции, но нужен version log

6. Поиск
Нужны следующие режимы поиска:
- exact search по function name
- exact/partial search по address
- search по behavior
- search по tags
- search по структурам
- search по hypotheses
- поиск связанных функций
- graph traversal на 1–2 hops
- дедупликация похожих записей
- reranking результатов

Требования к поиску:
- использовать SQLite FTS5 для полнотекстового поиска
- предусмотреть semantic search abstraction
- semantic search должен быть отключаемым и не должен ломать систему, если embeddings не настроены
- в MVP можно реализовать fallback: если embeddings отсутствуют, система работает только на FTS5 + exact search + graph relations
- интерфейсы под semantic search должны быть предусмотрены с самого начала

7. Импорт
- На первом этапе данные может вставлять агент или пользователь
- Нужно предусмотреть JSON import/export
- Нужно предусмотреть будущий импорт всех функций из IDA
- Архитектура импорта должна быть расширяемой
- На MVP этапе можно сделать import pipeline с адаптером, куда позже будет добавлен IDA importer

8. API и MCP
Нужен локальный HTTP API для внешнего агента.
Также нужен MCP server для Opencode Desktop.

Через MCP/HTTP агент должен уметь:
- искать записи
- получать запись по id
- получать связанную информацию
- создавать запись
- обновлять запись
- добавлять evidence
- добавлять hypothesis
- искать связанные функции
- импортировать данные
- экспортировать данные
- запускать backup
- получать конфиг текущего проекта/подключения

Нужно предусмотреть режим настраиваемого поведения:
- автоматически сохранять вывод агента
- требовать подтверждение пользователя перед сохранением

9. Ограничения на сохранение мусора
Система должна защищаться от загрязнения базы:
- запретить безразмерное сохранение длинного сырого текста
- хранить псевдокод только как короткие фрагменты evidence
- разделять facts и hypotheses
- не использовать append-only текстовые summary как основную модель хранения
- сохранять нормализованные записи
- валидировать обязательные поля
- валидировать максимальный размер полей

10. GUI
После backend и MCP нужен web GUI для ежедневного использования.
GUI должен позволять:
- первичную настройку приложения
- создание/выбор проекта
- настройку базы
- настройку MCP/API
- просмотр функций
- просмотр структур
- просмотр гипотез
- редактирование карточек
- просмотр истории версий
- поиск
- просмотр графа связей
- импорт/export JSON
- backup/restore
- мастер первого запуска

GUI должен быть простым в развёртывании и без тяжёлого frontend-стека, если это возможно.

11. Конфигурация и развёртывание
Нужно обеспечить простой запуск:
- пользователь запускает приложение
- настраивает базу
- создаёт проект
- получает конфиг для подключения MCP
- начинает пользоваться

В MVP GUI не обязателен, но конфигурация должна быть понятной через config file.
Нужен первый-run setup wizard либо через CLI/config, либо позже через GUI.
Нужна генерация конфигурации подключения для агента.

12. Архитектурные требования
Предложи минималистичную архитектуру с упором на надёжность и простоту развёртывания.
Предпочтительно:
- Python
- SQLite
- встроенный или минимальный web backend
- минимальное количество внешних библиотек
- чёткое разделение слоёв:
  - storage
  - domain models
  - services
  - search
  - MCP/API
  - GUI
  - config

14. Важные проектные решения
Нужно учитывать:
- version history видна в GUI, но агенту обычно отдаётся только актуальная версия
- псевдокод целиком хранить не надо
- related functions ищутся агентом по необходимости
- пользователь может вставлять и править данные вручную
- система полностью локальная
- сейчас таргет только Windows

15. Дополнительное требование
Если есть несколько разумных вариантов реализации, выбирай тот, который:
- проще развернуть локально
- требует меньше зависимостей
- понятнее поддерживать
- лучше работает offline

Не предлагай сложную микросервисную архитектуру, Docker-first дизайн, внешние базы данных или облачные зависимости, если без них можно обойтись.

Правила идентичности и дедупликации:
- canonical uniqueness для function: (project_id, binary_id, function_id)
- address в рамках binary_id должен быть уникальным или помечаться как conflict
- похожие записи не должны автоматически merge-иться
- система должна лишь помечать возможные дубликаты и ранжировать их

MVP не должен включать:
- тяжёлый frontend framework
- сложную auth-систему
- внешнюю векторную БД
- обязательные embeddings
- cloud sync
- полноценный IDA plugin
- автоматический merge conflicting records
